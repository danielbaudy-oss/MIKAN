MIKAN Punch Clock System

Legal Compliance Summary

Prepared for MIKAN Restaurant, Barcelona — April 2026

Legal Context

Spain's mandatory time recording law (Real Decreto-ley 8/2019), updated with new digital requirements expected to be fully enforceable in 2026, requires all companies to use a digital time-tracking system that meets specific technical standards. There is no official certification or homologation required — the law sets technical requirements, not a certification scheme. Any software that meets these requirements is compliant.

Compliance Checklist

✅ Digital System (Not Paper or Spreadsheet)

MIKAN runs on PostgreSQL (hosted by Supabase), a proper relational database. The frontend is a web application hosted on GitHub Pages. Paper sign-in sheets and Excel/Google Sheets spreadsheets are explicitly non-compliant under the updated regulation — MIKAN is neither.

✅ Remote Accessibility

The system is web-based and accessible from any device with a browser and internet connection. Employees clock in/out from their phones or a shared tablet. No software installation required.

✅ Inalterability / Tamper-Proof Audit Trail

Every database operation (insert, update, delete) is automatically logged in a dedicated audit_log table via PostgreSQL triggers. This logging cannot be bypassed from the application layer. Each audit entry records: the table affected, the record ID, the action performed, the complete old and new data as JSON, and a timestamp. Punch records are never physically deleted — they are soft-deleted with a mandatory reason and timestamp. Edits record the original time and the reason for correction.

✅ GPS Positioning

Every clock-in and clock-out captures the employee's GPS coordinates (latitude and longitude) via the browser's Geolocation API. Coordinates are stored alongside the punch record in the database. If the employee denies location access, the punch still records but without GPS data.

✅ Immediate Accessibility for Inspectors

A read-only database view (inspector_report) provides real-time access to all punch records including corrections, deletions, GPS coordinates, and timestamps. This view can be queried at any time. An inspector can be granted read-only access to the Supabase dashboard on demand, providing immediate and complete visibility into all records.

✅ 4-Year Data Retention

All data is stored in PostgreSQL with no automatic deletion or expiration. The audit log is append-only and retains records indefinitely. The database is hosted on Supabase's EU infrastructure (Frankfurt region available) with automatic backups.

✅ Employee Records, Manager Approves

Employees register their own clock-in/out times using a personal 4-digit PIN. The admin panel allows managers to review all records, approve or reject time-off requests, and oversee corrections. Any modification to a recorded entry is documented with a reason and logged in the audit trail.

✅ Not Biometric

Authentication uses a 4-digit PIN — not fingerprints, facial recognition, or any biometric data. This avoids the data protection concerns raised by recent Spanish case law and AEPD rulings regarding biometric time-tracking systems.

Bottom line: No software in Spain carries a government seal or certification for time tracking. What matters is meeting the technical requirements set by the regulation. MIKAN meets all of them: proper database (not a spreadsheet), complete tamper-proof audit trail, GPS records, immediate inspector access, 4-year retention, and documented correction workflows. This system is already more compliant than what the vast majority of small hospitality businesses in Spain are currently using.

Technical Details

Database: PostgreSQL via Supabase (EU-hosted)
Frontend: Static HTML/JS hosted on GitHub Pages
Authentication: PIN-based (no Google account required)
Audit mechanism: PostgreSQL AFTER triggers on all tables → audit_log table
Data format: All records stored with UTC timestamps, dates in ISO 8601 format
Punch deletion: Soft-delete only (is_deleted flag + reason + timestamp)
GPS: Browser Geolocation API, coordinates stored per punch record
